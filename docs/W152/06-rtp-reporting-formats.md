**TL;DR**  No two jurisdictions share the same reporting schema. Denmark and Italy demand real-time or near-real-time transactional XML/binary feeds; the UK, MGA and Ontario collect aggregated financial and liability data through web portals or SFTP; New Jersey uses Excel workbooks emailed monthly. An operator-agnostic engine must emit both granular session events (for Denmark/Italy) and rolled-up GGY/player-fund summaries (for UK/MGA/Ontario), with jurisdiction-specific adapters handling transport and format.**Key Findings*** **UKGC – Quarterly portal returns, not files.** Returns are submitted via the eServices web portal (not XML/CSV). The Remote Casino, Betting and Bingo return covers GGY, self-exclusions, customer-interaction counts and premises data, due 28 days after quarter-end (UK Gambling Commission, 2024). RTP is monitored separately by operators using the Commission’s tolerance methodology (UK Gambling Commission, 2025). Compliance data must be retained for at least five years after a customer relationship ends (UK Gambling Commission, 2025).* **MGA – Monthly player-funds web form.** B2C licensees file a Player Funds Report through the Licensee Portal by the 20th of the following month, declaring month-end player balances, jackpot liabilities and bank/PSP account balances, backed by screenshots and statements (MGA, 2024). New Annual/Interim Financial Reports apply from 2025 (BDO Malta, 2025). Player-interaction records must be kept for at least two years; AML records for five, extendable to ten (MGA Player Protection Directive, 2018).* **Italy ADM – Real-time binary/SOGEI integration.** The public PGAD protocol defines fixed-width binary messages (not XML/JSON) for skill and fixed-odds games, with fields such as Game Session ID, Participation ID, wager amount and timestamp (ADM, 2011). Remote casino/slots operate via real-time SOGEI central-system integration; every bet and win is recorded live (Chambers Gaming Law Italy, 2025). Transaction records must be retained for up to ten years under AML rules (Chambers Gaming Law Italy, 2025).* **Ontario AGCO/iGO – Standards-based logging and financial files.** The Registrar’s Standards mandate logging of unique player ID, game ID, session times, wager, outcome, progressive contributions and balances (AGCO, 2024). Operators submit daily/monthly GGR/NGR files over secure SFTP/SharePoint and must provide audited financial statements within 120 days and a GGR schedule by 15 May (PwC Ontario, 2025).* **New Jersey DGE – Excel workbook email.** The Monthly Gross Revenue Report (DGE-101) is an Excel file listing authorized units, win/loss, drop/handle and win percentage, attested by a digital signature and emailed to `ofi@njdge.org` by the 10th of the month (NJ DGE, 2023). Internet gaming uses DGE-105 and promotional credits use DGE-170.* **Denmark DGA – XML per-session schema on SAFE.** Casino games are reported in `KasinospilPrSessionStruktur` (XML with XSD), capturing player ID, session ID, game product, stake, win (ex-jackpot), number of moves and currency. Files must be placed on the operator’s SAFE storage so no transaction is older than five minutes; an End Of Day summary is due by 04:00 UTC (Danish Gambling Authority, 2025).* **Germany GGL – Safe Server / LUGAS file drops.** Licensed operators must deposit data on a “Safe Server” that the GGL accesses via its Auswertesystem. The exact schema is restricted to licensees, but the system feeds the LUGAS cross-operator deposit-limit and activity files (Spielregeln.de, 2024).* **Netherlands KSA – CRUKS + CDB, no public game-event schema.** Operators must integrate with CRUKS (BSN-based self-exclusion) and maintain a Dutch-based Control Data Bank (CDB) for regulatory monitoring (GBG/KSA, 2022). A standardized per-spin XML/CSV schema analogous to Denmark’s is not publicly documented.**Contradictions / Open Questions*** **Granularity split.** Denmark and Italy require session-level or bet-level detail; the UK, MGA and Ontario are satisfied with aggregated GGY and liability figures. An engine must support both streams without double-reporting.* **Germany’s black box.** The GGL Safe Server technical guideline and file format are not public—they are shared only with applicants. Public sources confirm the existence of the requirement but not whether the format is XML, CSV or proprietary binary.* **Netherlands telemetry gap.** Unlike Denmark, the Netherlands does not publish an XSD or fixed schema for slot-level reporting; regulatory oversight appears to rely on the CDB and periodic audit rather than standardized daily game-event files.**Practical Implications for Boki / host orchestrator**1. **Core event schema.** Emit a canonical event containing: `operator_id`, `player_pseudo_id`, `game_id`, `session_id`, `timestamp`, `wager`, `win`, `currency`, `game_type`, `jackpot_contribution`. This satisfies Denmark, Italy (via adapter), Ontario logging and NJ aggregation.2. **Adapter layer.** Build adapters for: (a) Denmark XML (XSD-validated SAFE upload); (b) Italy binary PGAD or SOGEI API wrapper; (c) NJ Excel generation; (d) UK/MGA/Ontario rollup APIs/SFTP; (e) Germany Safe Server file writer (format TBD upon licence application).3. **Retention policy.** Default to seven years for raw transaction logs to cover the strictest known requirement (Italy AML ten years for certain records, UK five years post-relationship).
