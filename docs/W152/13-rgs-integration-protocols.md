**TL;DR**  No universal RGS API standard exists; every major provider uses proprietary HTTPS/REST (or legacy SOAP) endpoints. The lowest-common-denominator surface a math-engine template must expose is: authenticated REST callbacks to an operator wallet (bet/debit, win/credit, rollback), idempotency via UUID, sub-200 ms synchronous response, and async event streaming for compliance. Open-source alternatives are immature, and GLI-19 (not GLI-31) is the de facto compliance baseline.---### Key Findings- **Protocol:** Operator-facing RGS APIs are HTTPS/REST with JSON today; legacy systems (e.g., Vendor A WagerWorks) used SOAP (WagerWorks, 2010; Stake Engine, 2025). Real-time game events may stream over WebSocket, but wallet integration is request–response REST (Capermint, 2026).- **Authentication:** No single standard. CasinoWebScripts uses HMAC‑SHA256 with a shared API Encryption Key (CasinoWebScripts, 2025). Hub88 uses RSA‑SHA256 request signing (Hub88, 2025). Capermint cites JWT (Capermint, 2026). A pluggable math engine must support multiple auth schemes.- **Bet/Win Lifecycle:** The canonical operator contract exposes balance query and adjustment endpoints—`getbalance` + `balance_adj` (bet/win/cancel) (CasinoWebScripts, 2025). Stake Engine uses `/wallet/play` and `/wallet/end-round` (Stake Engine, 2025). Hub88 separates bet, win, and rollback calls (Hub88, 2025). All require atomic debit‑before‑RNG and credit‑in‑same‑tx (Capermint, 2026).- **Idempotency & Retry:** Every source mandates idempotency via UUID (`uniqid`, `transaction_uuid`, `request_uuid`). Retry semantics differ: Hub88 retries 3× with 1 s timeout (Hub88, 2025); CasinoWebScripts retries once after 5 s then queues (CasinoWebScripts, 2025).- **Latency Budget:** Enterprise RGS architectures target sub‑50 ms per round (Capermint, 2026). In practice, operator wallet SLAs are sub‑200 ms; exceeding this triggers cascading retries and player churn.- **Free Spins & Bonus State:** Bonus mechanics are externalized via provider promo APIs. BetConstruct defines `FSWithdraw`/`FSDeposit` (BetConstruct, 2022). Yggdrasil BOOST and Vendor E Enhance grant free spins via back‑office/API (Yggdrasil, 2026; Vendor E, 2026). The math engine should accept a promo token that suppresses real‑balance debit.- **Shared Jackpots:** Jackpot wins are usually injected by a separate async service and reported via a distinct `JackpotDeposit` call or added to the win payload (BetConstruct, 2022; Capermint, 2026).- **Provider Public Specs:** Deep technical specs are not publicly available for most tier‑1 providers. Vendor E and Relax Gaming publish only high‑level integration guides (Vendor E, 2026; Relax Gaming, 2026). Yggdrasil GATI is a studio‑facing toolkit, not an operator API (Yggdrasil, 2026). Vendor B OpenGaming and Vendor A PlayDigital expose aggregation portals but no open API documentation (Vendor B, 2022; Vendor A, 2024). Hacksaw Gaming’s OpenRGS™ is proprietary (Hacksaw, 2025). OneTouch is consumed via aggregators like Hub88 (Hub88, 2026).- **Compliance & Standards:** GLI‑19 is the actual standard for interactive‑gaming / RGS certification (GLI, 2024). GLI‑31 is Electronic Raffle Systems, not RGS (GLI, 2015). There is no mature open‑source RGS standard; `dvs999/RGS` is experimental (GitHub, 2020) and Open Game System targets web‑game features, not iGaming wallets (OGS, 2024).---### Contradictions / Open Questions- **GLI‑31 vs GLI‑19:** Industry commentary often conflates the two. GLI‑31 covers raffles; GLI‑19 covers remote gaming systems. A math engine must be certified to GLI‑19.- **Latency:** Capermint’s 48 ms round‑trip is an idealized internal microservice benchmark. Real‑world aggregator hops (Hub88, SoftSwiss) regularly exceed 100 ms. It is unresolved whether sub‑50 ms is achievable end‑to‑end through multiple provider tiers.- **Auth divergence:** HMAC, JWT, and RSA‑SHA256 are all actively used. No provider is moving toward a unified scheme, forcing adapters.---### Practical Implications- Build the math engine as a **stateless service** accepting a round context (player, bet, promo token, jackpot contribution).- Expose a **canonical wallet adapter interface**: `debit`, `credit`, `rollback`, each idempotent with a UUID.- Target **<100 ms response** to the aggregator, emitting async compliance events.- Implement **auth as pluggable middleware** (HMAC, JWT, RSA) rather than a hardcoded scheme.- Do not rely on open‑source RGS frameworks for production; engage a **GLI‑19 accredited lab** early.---### Sources1.
