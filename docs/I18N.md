# I18N — Contributor Guide

> **Module:** `web/studio/src/i18n/`
> **Bootstrap wave:** W208 Faza 400.1 (en / es / pt / de)
> **Tests:** `web/studio/tests/i18n-w208.test.ts` + `locale-switcher.test.ts`

This is the canonical contributor guide for the Studio's
internationalization layer. The companion end-user / operator-facing
documentation lives at `web/docs/content/14-i18n.md`.

## Why we built our own

We deliberately avoided i18next / FormatJS / lingui because:

1. **Cert paper trail.** The active locale and the exact translation
   strings shown to a player need to be reproducible from the IR +
   operator-package metadata. Owning the loader gives us a stable
   audit hook.
2. **Bundle size.** Zero-dependency vanilla-TS i18n is ~3 kB gzipped
   for the runtime + ~4 kB per locale dict. i18next + ICU formatters
   is ~25 kB.
3. **Type safety.** `TranslationKey` is a union of literal strings
   generated from `en.ts`, giving compile-time autocomplete and
   `noImplicitAny` enforcement on every `t('…')` call.
4. **No magic.** Vite-native ES modules; no XHR fetch, no
   asynchronous-init footgun on first render.

## Public API cheat sheet

```ts
import {
  t,                       // (key, params?) => string + helpers
  setLocale,               // ('es') => 'es', persists + fires listeners
  getCurrentLocale,        // () => 'en'
  loadLocale,              // async hook (no-op today, async-ready)
  getSupportedLocales,     // readonly ['en','es','pt','de']
  getLocaleDisplay,        // (loc) => { flag, name }
  onLocaleChange,          // (cb) => unsubscribe
} from '../i18n/index.js';

t('nav.build');                                    // 'BUILD'
t('errors.validation', { reason: 'bad input' });    // 'Validation failed: bad input'
t.plural('items', 5);                              // Intl.PluralRules
t.num(1234.5);                                     // '1,234.5' / '1.234,5'
t.currency(99.5, 'USD');                           // '$99.50' / '99,50 $'
t.date(new Date(), { dateStyle: 'long' });         // locale-aware

import { isRtlLocale, directionFor } from '../i18n/rtl.js';
isRtlLocale('ar');  // true
directionFor('en'); // 'ltr'
```

## Adding a new translation key

1. Add to `en.ts` under the right namespace
   (`nav` / `actions` / `symbols` / `metrics` / `modals` / `errors` /
   `tooltips` / `producer`).
2. Mirror the key in `es.ts`, `pt.ts`, `de.ts` with localized values.
3. TypeScript will refuse compilation if any locale file is missing
   the key (the `TranslationDict` type enforces parity).
4. `npx vitest run tests/i18n-w208.test.ts` — parity matrix double-checks.

## Adding a new locale

1. **Pick a code.** Two-letter ISO 639-1 preferred (`fr`, `it`, `ja`).
   Use regional suffix only when it matters (`pt-BR` already covered
   by the `pt` entry).
2. **Create the dict.** Copy `en.ts` to `locales/<code>.ts`. Translate.
   Keep the same ordering and key set.
3. **Register.** In `types.ts`:
   - Extend the `Locale` union.
   - Append to `SUPPORTED_LOCALES`.
   - Add `LOCALE_DISPLAY[<code>] = { flag, name }` (use the country
     emoji that maps to the language).
4. **Wire.** In `index.ts`:
   - `import { <code> } from './locales/<code>.js';`
   - `DICTS[<code>] = <code>;`
5. **RTL?** If yes, ensure the base tag is in `RTL_BASE_TAGS`
   (`rtl.ts`).
6. **Test parity.** Add the new code into the parity-matrix loop in
   `tests/i18n-w208.test.ts`.
7. **Visual QA.** Launch the studio, switch via the LocaleSwitcher,
   walk the BUILD / COMPOSE / CATALOG / PLAY tabs.

## Naming conventions

- **Namespaces:** keep the 8 documented namespaces stable. Add new
  ones only with reviewer approval — they show up in PAR docs and
  changing them is a paper-trail event.
- **Case:** `dot.snake_case`. No camelCase, no kebab-case.
- **Placeholders:** `{name}` ICU-flat. Names match across locales 1:1.
- **Punctuation:** preserve trailing periods / question marks in the
  source; the translation should match the source's punctuation style
  unless a target language demands otherwise (e.g. Spanish opening
  `¿`).
- **Plurals:** `<base>.one`, `<base>.other` (plus `few` / `many` /
  `zero` where the language requires them). Call via
  `t.plural(base, count)`. Plural categories follow CLDR; verify with
  `Intl.PluralRules(<code>).resolvedOptions().pluralCategories`.

## Translator workflow

1. Translator edits `web/studio/src/i18n/locales/<code>.ts` directly
   (it's a typed module, but the structure is dictionary-flat — feels
   like JSON with autocomplete).
2. Reviewer runs `npx tsc --noEmit -p web/studio/tsconfig.json` —
   catches any missing keys.
3. Reviewer runs `npx vitest run tests/i18n-w208.test.ts` — parity
   matrix + empty-string assertion.
4. Visual QA: `npm run dev` from `web/studio/`, flip the locale,
   walk all tabs.
5. Quality bar: no obvious machine-translation artifacts (e.g.
   verb-tense mismatch, untranslated brand-loanwords where they
   should be translated, calques from English idioms). The
   acceptance reviewer should be fluent or have a fluent reviewer
   sign off.

## RTL plan

Today: en / es / pt / de — all LTR. The `rtl.ts` module recognises
Arabic / Hebrew / Persian / Urdu / Sindhi / Pashto / Uyghur / Yiddish
base tags and will flip `<html dir>` + CSS variables automatically the
moment a translation for one of those codes is registered.

CSS authoring rules to keep RTL clean:

- Use `text-align: var(--text-align-start, left)` rather than literal
  `left`.
- Use logical properties (`padding-inline-start`, `margin-inline-end`)
  where browser support allows.
- Audit chevron / arrow glyphs — they need horizontal flipping under
  `dir="rtl"`.
- Mixed-direction text (digits inside RTL strings) is handled
  natively by the browser; don't fight it.

## Persistence & boot order

- The selected locale is persisted under `slot-math-studio.locale` in
  `localStorage`.
- On module import, the i18n core reads that value and applies it
  (sync). This means `getCurrentLocale()` reflects user choice
  immediately, with no async hop.
- If `localStorage` is unavailable (file://, private mode, blocked),
  the module silently falls back to `en`. No throws.

## Open questions for follow-up waves

- **Auto-detection from `navigator.language`** on first boot (only if
  not yet persisted). Currently we default to `en`. Boki may want
  region-aware boot in a Faza 400.2 wave.
- **Lazy-loaded dictionaries.** Today all 4 locales ship in the main
  bundle (~16 kB). With 10+ locales we should switch `loadLocale`
  to dynamic `import()`.
- **Pluralization-rich keys.** Today only `count` plurals are
  modelled. ICU's `select` (gendered nouns) is not — add when needed.
- **Translation memory export.** A `npm run i18n:export` task that
  emits XLIFF for hand-off to professional LSP vendors.
