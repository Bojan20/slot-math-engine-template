# RNG full-suite runner — How to populate the audit-grade reports

The Node.js baseline (`scripts/rng-quality.mjs` + `reports/rng/*-nist-baseline.json`)
catches first-order quality defects. **Audit submissions** (UKGC, MGA,
GLI-19) require the full external batteries. This document is the exact
recipe a CI operator runs to populate the missing artefacts.

## Required tooling

| Tool                | Install (macOS)                                | Install (Linux)                   |
|---------------------|------------------------------------------------|-----------------------------------|
| TestU01 (BigCrush)  | `brew install testu01` (community tap) or build from source | `apt install testu01` |
| NIST SP 800-22      | Build from <https://csrc.nist.gov/projects/random-bit-generation/documentation-and-software> | Same |
| PractRand           | `brew install practrand`                       | Build from <http://pracrand.sourceforge.net/> |

If a tool is unavailable, capture the install/build failure and link it
in the submission narrative under "tooling availability".

## Artefacts to populate

Each backend × battery produces one canonical artefact. Populate the
slots below; the `audit-kit` CI job picks them up automatically.

| Slot                                              | Battery        | Backend          |
|---------------------------------------------------|----------------|------------------|
| `reports/rng/pcg64-bigcrush.txt`                  | TestU01        | pcg64            |
| `reports/rng/xoshiro256ss-bigcrush.txt`           | TestU01        | xoshiro256ss     |
| `reports/rng/philox4x32-bigcrush.txt`             | TestU01        | philox4x32       |
| `reports/rng/pcg64-nist-full.json`                | NIST SP 800-22 | pcg64            |
| `reports/rng/xoshiro256ss-nist-full.json`         | NIST SP 800-22 | xoshiro256ss     |
| `reports/rng/philox4x32-nist-full.json`           | NIST SP 800-22 | philox4x32       |
| `reports/rng/pcg64-practrand-2e38.txt`            | PractRand      | pcg64            |
| `reports/rng/xoshiro256ss-practrand-2e38.txt`     | PractRand      | xoshiro256ss     |
| `reports/rng/philox4x32-practrand-2e38.txt`       | PractRand      | philox4x32       |

`mulberry32` is intentionally **not** in the table — it is retained only
for TS↔Rust byte-for-byte parity and is permitted to fail external
batteries. See `docs/rng.md` for the policy.

## Generating raw byte streams

All three batteries read raw bytes from stdin. The engine ships a small
dumper via `node scripts/rng-quality.mjs --dump <backend> <bytes>` →
binary on stdout.

### TestU01 BigCrush (≈ 1 hour per backend)

```bash
node scripts/rng-quality.mjs --dump pcg64 4000000000 \
  | TestU01 -b bbattery_BigCrush -i stdin \
  > reports/rng/pcg64-bigcrush.txt
```

### NIST SP 800-22 (≈ 15 min per backend)

The official `assess` tool wants a `.dat` file of 10⁸ bits. Generate, run,
then convert the report:

```bash
node scripts/rng-quality.mjs --dump pcg64 12500000 > /tmp/pcg64-100m.bin
( cd /opt/nistsp80022 && ./assess 100000000 < /tmp/pcg64-100m.bin )
# move and convert finalAnalysisReport.txt → JSON via scripts/nist-to-json.mjs
node scripts/nist-to-json.mjs /opt/nistsp80022/experiments/AlgorithmTesting/finalAnalysisReport.txt \
  > reports/rng/pcg64-nist-full.json
```

### PractRand 2³⁸ bytes (≈ 30 minutes per backend on M-series)

```bash
node scripts/rng-quality.mjs --dump pcg64 274877906944 \
  | RNG_test stdin -tlmax 2^38 -multithreaded \
  > reports/rng/pcg64-practrand-2e38.txt
```

## Acceptance bars

| Battery        | Acceptance                                        |
|----------------|---------------------------------------------------|
| BigCrush       | 0 failures across all 160 tests at α = 0.01       |
| NIST full      | 188 / 188 sub-tests pass at α = 0.01              |
| PractRand 2³⁸  | No failures up to and including 2³⁸ bytes (256 GB)|

`pcg64` (production default) MUST clear all three; any failure on `pcg64`
is a release blocker. The other two backends have the same bar.

## Acceptance bookkeeping

Once you have populated the slots, update three places:

1. `docs/compliance.md` § GLI-11 clause 4.1 — flip the row from ⚠️ to ✅
   and reference the new artefact paths.
2. `SLOT_ENGINE_MASTER_TODO.md` P0 plug-list item #3 — flip ❌ → ✅ with
   per-backend pass/fail line and link to `reports/rng/INDEX.md`.
3. `reports/rng/INDEX.md` (auto-regenerated by `scripts/rng-quality.mjs`)
   — re-run the script so the table picks up your new artefact links.

## Why the Node baseline still ships

The 5-test Node baseline runs in CI on every PR (≈ 2 s/backend) and
catches catastrophic regressions: a one-line bit-flip bug in a backend's
`next_u64` would fail Monobit immediately. The external batteries run
weekly / per-release because they take an hour-plus per backend; the
Node baseline is the always-on tripwire.
